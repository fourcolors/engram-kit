What Opus consistently notices

  • Commit SHAs as ground truth — every rep cross-checks against local_git_anchor.json SHAs (1cbd734, 63cc4a5, d719f3f, 7afd378, fe4135c). When a SHA matches a session claim, score is "strong"; when the answer cites a SHA absent from the anchor, judge flags confabulation.
  • Session IDs as fingerprints — judge uses ses_3367fd033ffe, ses_336933258, ses_317004300ffeKOtF1Jzo4utYCR, ses_30d2c1873ffebsIycnvqLgc1D2 to route claims to sources. Misattributed session IDs are consistently caught (rep1 pure R19 subagent-recap vs user-authored work).
  • Verbatim user quotes — "we don't do mcp right now, cli and agent as skill", "lets make it real time first. Focus on 12. That the killer feature", "Forget retention forgetting for a while". When recovered, factual_grounding is "strong".
  • Slice cardinality — judge counts JSONL files (407, 711, 742, 2894, 3197, 3663) and flags "cold start" / "no activity" claims against them. One answer's "3222 sessions" was caught as off-by-many-hundreds.
  • Temporal boundary checks — reference_dt vs slice edge is recomputed every rep. Wrong-date drift (reporting 2026-04-16/17 when ref is 2026-03-12 or 2026-03-21) is the single most consistent failure mode the judge surfaces.

Judge self-contradiction failure modes

  • Coherence axis gets overweighted when factual is strong — in rep1 syke R11, coherence subscores default to "strong" matching the factual pattern, without distinct evidence-work.
  • Bug #9 contradiction handling — rep1 syke R19 notes Bug #9 was committed-fixed on Mar 20 but answer lists it as ongoing. Coherence drops to "partial" over this. Rep3 syke R19 catches the same issue but verdict stays "pass" in rep1 vs "partial" in rep3 despite identical evidence.
  • Paraphrase vs slice-cite — many reasoning fields paraphrase what the answer said rather than quoting slice text. Clean quote-citation only appears when verbatim user quotes are load-bearing. Most factual_grounding.support reasonings assert "all claims backed by slice evidence" without enumerating which.
  • Pure-condition double standard — rep1 pure R11 architecture vision gets "partial" with "honest disclaimer" as merit; rep3 pure R11 scores the same vision "partial" but downgrades for confidence framing. Uncertainty calibration is judged inconsistently.

Reasoning length as signal

  • Short reasoning (<400 chars) appears almost exclusively in "missed" subscores where the judge is dismissive. These short reasonings cluster on fail verdicts (zero R04 rep1/rep2, zero R07, zero R08, zero R19).
  • Long reasoning (>1500 chars) appears on pass and high-partial verdicts, especially factual_grounding.reasoning and summary fields.
  • Asymmetry: strong verdicts average longer reasoning than fails. The judge does real evidence work to justify passes but treats fails as self-evident.

Coherence-axis pathology

  • Stringified JSON cases — rep1 pure R01/R07/R08/R11/R14, rep1 syke R01/R08/R11 have coherence as a JSON-encoded string rather than a structured dict. Rep1 pure R17, rep2 syke R08, rep3 zero R11 have it as proper object. Same judge, inconsistent output shape.
  • Subcategory rubber-stamping — when coherence is strong, all four sub-axes (cross_harness_braid, cross_session_consistency, artifact_routing_consistency, contradiction_handling) tend to be strong with nearly identical reasoning boilerplate. Real differentiated work only appears on partial/missed scores.
  • cross_harness_braid under-engaged — when only claude-code data exists, judge scores "strong" for "correctly focusing on available harness". That's absence-of-work scored positive.
  • Contradiction handling is mostly "no contradictions detected" — across 45 files, genuine contradiction-surfacing appears in <10 cases. Most contradiction_handling scores default to strong by vacuity.

Syke condition is where all the motion is. Opus converts 5 of 7 GPT-5.4 "partial" verdicts to "pass". Pure shifts modestly. Zero is invariant — genuine failures stay failures. The GPT-5.4 "partial attractor" on syke dissolves under Opus.

Axis-level shifts

  • Continuity is the axis that moves most. GPT-5.4 repeatedly docked continuity for "not the freshest cutoff-time state" even when the answer recovered the dominant active thread. Opus reads continuity as "would this restart the user's work" rather than "is every git-anchor commit surfaced".
  • Factual grounding moves less — where both judges see fabrication, both call it. Where they see solid evidence chains, both recognize them — but Opus attaches the verbatim quote that anchors the claim.
  • Coherence appears in Opus as a first-class axis with subcategories (cross_harness_braid, cross_session_consistency, artifact_routing_consistency, contradiction_handling). GPT-5.4 lacks this axis in 7 of the read cells — it folds coherence into continuity prose.

Reasoning depth & evidence density

  • Mean reasoning length per axis: GPT-5.4 ≈ 120 words per axis, no subcategories. Opus ≈ 350-500 words per axis with 3-5 scored subcategories.
  • Opus cites specific artifacts at ~3-4× the rate of GPT-5.4. Routine: session IDs (ses_3367fd033ffe), commit SHAs (1cbd734), verbatim user quotes, test output strings flagged as unverifiable. GPT-5.4 describes; Opus quotes.
  • Hedging: GPT-5.4 hedges in continuity narrative ("not fully current", "materially incomplete"). Opus hedges in structured uncertainty_calibration cells and awards "strong" while still noting a gap.

Stylistic / cognitive differences

  • Opus catches source-attribution bugs GPT-5.4 misses — pure/R19: Opus notices the answer cited a subagent meta-query response as if it were user work; GPT-5.4 scored the same cell pass.
  • Opus catches fabrication more aggressively even on answers both judges like — syke/R15: Opus flagged "observe-phase2 branch", "18 commits ahead", "28 passed in 8.13s" as "not found anywhere in the slice or git anchor". GPT-5.4 accepted the same numbers.
  • Opus does not show the partial-attractor bias GPT-5.4 had. Strong evidence + right thread restored → pass, even if git-anchor completeness is imperfect. GPT-5.4 downweighted for residual completeness almost mechanically.
  • Opus overweights source provenance (where did this specific claim come from). That matters for benchmark interpretation.

Bottom line — calibrate, don't choose

  • Keep both judges and calibrate against each other. Opus and GPT-5.4 are not competing versions of one instrument; they are two different instruments whose convergences and divergences carry information. The convergence set is the stable signal. The divergence set is where we investigate the cell further rather than pick a winner.
  • Opus is sharper on provenance (catches subagent-recap-as-user-work, fabricated test counts, missing SHAs in git anchor). GPT-5.4 is stricter on completeness (downweights for residual git-anchor gaps even when the dominant thread is restored). Neither is "right"; they're weighting different parts of the useful-answer definition.
  • Trustable only where both judges agree — zero-condition fails (both converge), pure-passes where both judges award pass with cited evidence, and pure-bad cells where both judges mark fail or partial. This last set is the real memory signal (see below).
  • Not claimable from this run: absolute pass-rate comparisons, architecture ordering beyond pure vs syke on the 3-cell convergence zone.

Taxonomy of flip mechanisms

  1. Authoritative-source instability. When git, session transcripts, and memex disagree about "the last thread", each rep picks a different anchor. Single biggest flip driver (syke R14, pure R14, zero R09).
  2. Cascade depth of a single flaw. One observed problem can stay quarantined in one sub-axis (rep1 syke R19) or propagate into overall verdict (rep2/rep3 syke R19). No stable policy on blast radius.
  3. Hedge tolerance. Disclaimers ("Apr 17 is empty", "no persisted memory, best-effort") read as honest scoping in some reps, boundary violation in others.
  4. Virtue / vice framing. Surfacing unresolved tension graded as "good contradiction handling" or "stale residue" depending on sub-axis routing.
  5. Sub-axis routing roulette. When reps agree on a gap, they may assign the "missed" score to different sub-axes (pure R11) — axis-level flips without overall-verdict flips.

  • R09 is the single cleanest memory-value signal. Pure fails in both judges; syke passes in both. The task is a numbered thread-map with user-maintained status transitions — exactly the kind of structured work-graph a memex is designed to hold.
  • R02, R07, R14 are Opus-only uplifts. Opus sees syke restore the thread and awards pass; GPT-5.4 keeps syke at partial or fail. These are calibration cells — the Opus-vs-GPT-5.4 delta is load-bearing on whether memory value is recognized.
  • R04, R05, R15 are the hard floor. Both judges agree no condition recovers them — R04 is environment (time anchor), R05 is memex-staleness inside syke, R15 is a completeness task where even mining the slice doesn't fully reconstruct.
  • R08 is the thin-bootstrap case. Pure fails both, syke only partially recovers under Opus. This is where a memex is supposed to carry state forward and the snapshot wasn't dense enough.
  • R18 shows zero can win too. Zero uplifts to pass under Opus on R18 — adapter mining alone recovered enough. This is a reminder that zero is not strictly inferior; it just has a much narrower working range.

What to do about the 10 probes pure passes on

These are tasks recoverable from the slice itself without memory. They are not useless — they test that the memory system doesn't hurt an answer that the agent could already give. Track them as the "do no harm" set. Any condition that regresses pure-passes to partial/fail on these cells is actively damaging.

Proposed calibrated schema — three axes, seven sub-axes

factual_grounding
  • support — is each claim backed by specific slice/git evidence?
  • boundedness — are claims kept inside the bounded evidence (no out-of-slice, no wall-clock leak)?
  • uncertainty_calibration — are hedges honest; is overclaim avoided?
continuity
  • thread_selection (merges active_thread_selection + salience_relevance) — is the right current thread surfaced, with right relative weight?
  • state_transition_tracking — does the answer track what changed since the last cycle: what's now done, what's now live, what's now residue? This was the top-defect-load subcategory; it carries most of the real signal.
  • forgetting_residue_control — is stale state labeled stale, not presented as current?
  • continuation_value — would this answer let the user resume useful work?
coherence
  • internal_consistency (merges cross_session_consistency + artifact_routing_consistency) — do the claims braid consistently across sessions, files, timestamps?
  • contradiction_handling — when memex and git (or two sessions) disagree, does the answer surface the contradiction rather than pick one silently?

Retire cross_harness_braid as a scored sub-axis. Fold it as a conditional note — only evaluated on multi-source probes. When only one harness has data, scoring it as "strong" for "correctly focusing on available harness" is a vacuity score that masquerades as signal.

Why this shape

  • 7 sub-axes instead of 12. Reduces judge cognitive load — the main reason coherence subcategories rubber-stamp "strong" is that the judge is spreading attention too thin.
  • Keep the defect-load leaders unchanged. state_transition_tracking, active_thread_selection (now thread_selection), contradiction_handling, continuation_value carry 4 of the top 5 miss counts. These are the actual discriminators.
  • Merge only when semantically redundant. salience_relevance fires when active_thread_selection does; artifact_routing_consistency fires when cross_session_consistency does (judges routinely give them the same score). Merging is a data-driven consolidation, not a subjective simplification.
  • Retire the conditional-only sub-axis. cross_harness_braid is only meaningful on multi-source cells. Keep the concept as a probe-level tag rather than a sub-axis every cell gets scored on.

Validation path — don't ship the new schema without running it

  1. Apply the 7-sub-axis schema retroactively to the ab07 judge traces (we have the prose reasoning; most sub-axis scores can be reconstructed from it).
  2. Check whether the convergence set (pure-bad cells both judges agree on) gets sharper — fewer flip cells at the verdict level, more stable sub-axis agreement across reps.
  3. Run one small fresh benchmark pass with the new schema in the Pi judge tool, 1 rep each on GPT-5.4-mini and Opus, compare flip rate vs ab07's 37/68/32%.
  4. Only if the new schema reduces flip rate without dropping discriminator power, promote it for the 3×3 matrix.

Stable-fail catalogue as the real taxonomy

The 20 stable-fail cells decompose cleanly into 8 failure modes, and they cluster by condition:

  • environment-side: time-anchor bleed (R04 universal). Targeted by Apr 18 patch.
  • zero-architectural: thin-bootstrap self-report (R08, R09), refusal-to-mine (R15), out-of-slice hallucination (R04, R07, R08, R15, R19). Four distinct zero failure modes.
  • syke-architectural: stale-memex residue (R05), memex-anchored-too-early (R12), memex-denial-of-fresh-session (R16). All driven by the memex being load-bearing in the wrong direction.
  • pure-architectural: missed numbered-thread system (R09), attributed a subagent recap as user work (R19).
  • shared-architectural: implementation-state undersell (pure R11 + syke R12 + zero R11/R14/R17) — the hardest class; all three conditions fail because the slice itself was ambiguous about what was built vs planned.

Operational recommendation for the 3×3 matrix — calibration-first, architecture-later

  1. Run both judges and report both. Every run scored by GPT-5.4 and Opus in parallel. Report convergence (both agree) as the trustable signal; report divergence cells separately as calibration targets, not as ordering evidence. Do not pick one judge as canonical yet.
  2. Pure-fail consensus is the real progress ledger. The 9 pure-bad probes (R02, R04, R05, R07, R08, R09, R14, R15, R18) where both judges agree pure is not-pass are where the environment demands memory. Count how many of these a condition can lift to pass (in both judges) — that's the useful measurement. Don't inflate by including cells pure already passes.
  3. Do-no-harm set. The other 10 probes where pure passes — any architecture that regresses them to partial/fail is actively damaging, independent of its pass-rate on the pure-fail set. Track both numbers.
  4. Smoke R04 first. R04 is the environment-side time-anchor failure, stable across both judges and all three conditions. If the Apr 18 time-unify patch works, R04 should recover at least partial. If it doesn't, one cell will keep dragging every aggregate down.
  5. Validate the 7-sub-axis schema before the big run. Apply it retrospectively to ab07 traces and run one 1-rep pass under both judges with the new schema. Only ship it to the 3×3 if flip rate drops meaningfully without losing discriminator power. Architecture stays frozen during this work — we are calibrating the instrument, not the subject.
  6. Expect syke to still have a high flip rate. Syke surfaces more content, so judges have more to disagree about. Three reps per cell are load-bearing regardless of schema changes — report modal verdict plus full distribution, never just one verdict.

Why pure-fail focus, not syke-focus

Pure's failures reveal what the environment actually demands: longitudinal state, cross-session provenance, thread structure that the slice doesn't self-organize. These are the tasks Syke exists to carry. Reading the probe set through pure-fails instead of syke-passes flips the question from "how often did memory help?" to "on the tasks where memory is required, did this architecture deliver?" — which is the claim a memory product actually has to meet.

The framing from prior sessions (via syke): Syke's data is real longitudinal cross-harness memory about an evolving work graph, where truth is time-dependent and must be reconstructed from fragmented traces. Pure-fails are the probes where that reconstruction demand is explicit; syke-passes on pure-passes are not evidence of anything. This is why the pure-fail consensus set is the signal to optimize against.

What this probe did not answer

  • Whether the 3×3 (replay × eval-pack) matrix will show lower flip rate on repeat ask. This probe is judge-only; ask variance is unmeasured.
  • Whether new synthesis mutations (syke_guard, syke_checkpoint, syke_route) move any pure-fail consensus cells — those runs are queued, not yet evaluated under either judge.
