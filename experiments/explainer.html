01 What is ENGRAM?

A memory test for AI — with one strict rule: no peeking at the future.

Imagine a messy research folder that changes a little every day for two weeks — files appear, get rewritten, get moved into an archive/, get superseded by newer versions. ENGRAM hands an AI one daily snapshot at a time, in order, and asks it to build up a memory as it goes.

Later, it asks questions like "as of April 13, which version of the eval set is the canonical one?" — and the AI must answer as if it were standing on that day, citing the actual files as evidence, and honestly flagging what it can't be sure of. The grade rewards four things:

  • Telling current from stale — what's live now vs. superseded or archived.
  • Citing real evidence — pointing at actual files, not making things up.
  • No hindsight — never using knowledge from a day that hasn't happened yet.
  • Honest uncertainty — saying "I can't tell" when the file genuinely isn't there.

The whole thing runs through just two commands: update (here's today's snapshot — remember it) and answer (here's a question — reply with citations).

02 How our system works

agent-memory keeps one growing notebook of dated bullet-notes. Two passes read and write it.

① Observer pass  ·  runs on every daily snapshot

Reads the day's changes.txt (what changed) + manifest.tsv (which files are confirmed vs. approximate) and writes 1–5 dated notes about what's new, what moved, what got superseded.

./run.sh update STATE_DIR MEMORY_DIR
→

② MEMORY.md  ·  the notebook

One Markdown file. Notes grouped under Date: headers, tagged by priority. Grew to 151 lines over the 14 days — a living timeline of the folder.

🔴 critical · 🟡 fact · 🟢 minor · ✅ done
↓

③ Answer pass  ·  runs on every question

Re-reads the notebook up to the question's date, opens the relevant files from that day's snapshot, then replies with a direct answer, evidence paths, and an honest uncertainty note.

./run.sh answer STATE_DIR MEMORY_DIR QUESTION ANSWER
→

④ Answer JSON

answer · evidence_paths · memory_refs · uncertainty — the graded output.

the important nuance

The "intelligence" is a real model (Claude Sonnet) doing the observing and answering — run locally through the claude CLI, no API key. The notebook is just plain text you can open and read. Nothing is hidden in a database.

05 The key finding

The thing holding the score back was not the memory. It was how the answer pass fed itself the files.

In the stock version, when a question targeted a big file (a 30 KB eval set, a 19 KB sampling note), our code packed small files first into a size budget — so the big, important file never reached the model. Seeing only the filename, the model would shrug: "that file isn't available, I can't determine the answer." A real, present file, declared missing.

Stock — Apr 14 (scored 1/5)
Adapted — Apr 14 (scored 5/5)
"The content of S07_VS_LOCOMO_LIFEBENCH.md is not available in the provided file contents… The specific claims about how S07 differs from LoCoMo and LifeBench cannot be determined from the available evidence."
"S07 differs from LoCoMo and LifeBench in five structural ways each: S07 is cross-harness live work-graph reconstruction (claude-code, opencode, hermes…) vs. a single bounded dialogue; the output is a live work graph, not QA over annotated ground truth; provenance is scope-attribution, not speaker-attribution…"
what actually changed in v1
  • Let the model see what it cites — rank the day's files by relevance to the question (an explicit filename mention wins) and hand over the full bodies of the top matches.
  • Date-filter the notebook — only feed notes dated on or before the question's day. This closed the one hindsight leak (Apr 10).
  • Ban the false "unavailable" — tell the model: if a file's content is in front of you, use it; memory is a hint, not evidence.

One change paid three dividends: groundedness jumped, the false refusals vanished, and the 96-second timeout outlier disappeared — because a focused, relevant digest is actually smaller than the old "dump all the small files" approach.

06 Improving the memory itself

v1 fixed the reading side. The biggest remaining gains — and the most durable ones for a real, months-long deployment — are in the notebook itself.

Today the memory is excellent prose: the observer already caught the Apr-20 folder restructure, the approximate→exact flip, and supersession chains, all on its own. But prose is hard to query precisely, which is exactly where the two remaining weak answers (Apr 18, Apr 20) went wrong — they made attribution mistakes ("this file first introduced X" when an earlier file already had it; "all 80 files flipped status" when really 23 did). A richer memory structure would catch these.

MEMORY · #1
A per-file timeline, not just prose
What Alongside the bullet-notes, keep a structured record per artifact: first_seen, status changes (approximate→exact), archived, superseded_by.
Why Turns "when did X first appear / what replaced it" from a guess into a lookup.
Fixes The Apr 18 "first introduced" error — memory would know the R## scheme appeared days earlier.
MEMORY · #2
Reflection that preserves lineage
What When the notebook gets long, today's compression keeps "newest wins." Change it to preserve the dated chain (current Apr 10 → archived Apr 20) instead of collapsing to the latest state.
Why The sample (14 days) never triggered compression — but a real ENGRAM run spans weeks. Then "newest wins" would quietly erase the as-of-date truth and reintroduce hindsight errors.
Fixes Scales no-hindsight correctness to long horizons — the difference between a demo and a deployable system.
MEMORY · #3
A novelty / "first-appearance" ledger
What Explicitly log the first time each named entity (a namespace, an eval-set version) shows up, so any "new / first / introduced" claim can be checked against it.
Why "First/novel" claims are where the model over-reaches; the memory should be the authority that vetoes a false one.
ANSWER · v2
Finish the read-side cleanup
What A hard check that forbids saying "unavailable" about any file present in the manifest; a quick verify pass that re-grounds every count and "stated in FILE" claim; always show the full file list so "not in my digest" ≠ "not in the folder."
Fixes The last two laggards (Apr 18 → ~5, Apr 20 → ~5), targeting a near-perfect average.
bottom line

Your agent-memory discipline is genuinely strong — it built an accurate two-week timeline with zero hand-holding. v1 already took the system from 2.60 to 4.40 / 5 with no hindsight leaks, purely by fixing how it reads. The next ceiling-raiser is making the notebook structured enough to answer attribution questions and to stay honest over long horizons.
